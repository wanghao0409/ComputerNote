## 2.SSH协议

SSH 就是一个直接面向用户和应用程序的协议，它提供了安全的**远程登录**、命令执行**、**文件传输**等功能。

### 2.1 用户认证

服务器收到加密的用户凭证后，完成认证。

1. **密码认证**：
    - 客户端用服务器的公钥**加密**用户输入的密码，发给服务器。
    - 服务器用自己的**私钥**解密，验证密码是否正确。
2. **公钥认证（推荐，更安全 + 免密）**：
    - 用户事先把自己的**公钥**放在服务器的 `~/.ssh/authorized_keys` 中。
    - 客户端向服务器发送一个“我要用公钥登录”的请求。
    - 服务器生成一段随机数据，用客户端的公钥加密后发给客户端。
    - 客户端用自己本地的**私钥**解密，把解密结果发回给服务器。
    - 服务器验证结果是否正确。
    - 整个过程**密码永远不通过网络**，且私钥只在客户端本地，从不传输。
- **公钥认证**：这是一种更安全、更便捷的方式。你需要**事先生成一对密钥**（公钥和私钥），并将公钥放在服务器上。认证时，服务器会用公钥加密一个“挑战”发送给客户端，只有持有对应**私钥的客户端**才能正确解密并回应，从而证明身份[](https://cloud.tencent.com.cn/developer/article/2015563?from=15425&frompage=seopage)。这种方式可以有效防止暴力破解，并实现免密登录。

ssh-keygen -t ed25519 -C "your_email@example.com"


ssh -i /path/to/private_key user@hostname

传输层解决的是**端到端**的通信，关注数据是否完整、有序、无差错，以及是哪个应用程序（进程）在收发数据
## 一 . TCP协议（传输控制协议）
### TCP的核心特性

1. **面向连接**：通信前必须先通过“三次握手”建立一条虚拟连接，通信结束后通过“四次挥手”释放连接。
2. **可靠传输**：通过多种机制保证数据无差错、不丢失、不重复，且按序到达。
	- **校验和**：检测数据在传输中是否损坏。
    - **序列号**：为每个字节编号，用于排序和去重。
    - **确认应答**：接收方收到数据后必须回复确认（ACK），发送方收到确认才放心。
    - **超时重传**：发送后若长时间未收到确认，则判断数据丢失并重新发送。
3. **流量控制**：使用“滑动窗口”机制，让发送方根据接收方的处理能力调整发送速度，避免接收方缓冲区溢出。
4. **拥塞控制**：网络拥堵时，发送方主动降低发送速率，避免加剧网络瘫痪。常见算法包括慢启动、拥塞避免、快速重传等。
5. **基于字节流**：TCP将应用层传来的数据视为一连串无结构的字节流，它会自行切分成合适的报文段发送，而不是保持应用层的数据块边界。
6. **全双工通信**：连接双方可以同时发送和接收数据。
### 连接管理：三次握手与四次挥手

这是TCP最经典的机制。

#### 1. 三次握手 —— 建立连接

作用是同步双方的序列号和确认号，协商TCP参数，并确保双方都有收发能力。

- **第一次握手**：客户端 → 服务端，发送`SYN=1`，序列号`seq=x`（客户端进入 `SYN_SENT` 状态）
    
- **第二次握手**：服务端 → 客户端，回复`SYN=1, ACK=1`，确认号`ack=x+1`，自己的序列号`seq=y`（服务端进入 `SYN_RCVD` 状态）
    
- **第三次握手**：客户端 → 服务端，发送`ACK=1`，确认号`ack=y+1`，序列号`seq=x+1`（双方进入 `ESTABLISHED` 状态）
    

#### 2. 四次挥手 —— 断开连接

因为连接是双向的，所以每端都需要单独关闭自己的通道。

- **第一次挥手**：主动关闭方（如客户端）→ 对方，发送`FIN=1, seq=u`（主动方进入 `FIN_WAIT_1`）
    
- **第二次挥手**：被动关闭方 → 主动方，回复`ACK=1, ack=u+1, seq=v`（被动方进入 `CLOSE_WAIT`；主动方进入 `FIN_WAIT_2`）
    
- **第三次挥手**：被动关闭方发送完所有数据后 → 主动方，发送`FIN=1, ACK=1, seq=w, ack=u+1`（被动方进入 `LAST_ACK`）
    
- **第四次挥手**：主动关闭方 → 被动方，回复`ACK=1, seq=u+1, ack=w+1`（主动方进入 `TIME_WAIT` 等待2MSL后关闭；被动方收到后关闭）
### TCP报文段结构（简化版）

一个TCP报文段包含以下关键字段：

| 字段       | 说明                                                               |
| -------- | ---------------------------------------------------------------- |
| 源端口、目的端口 | 标识发送和接收的应用进程                                                     |
| 序列号（32位） | 当前报文段数据第一个字节的编号                                                  |
| 确认号（32位） | 期望收到对方下一个报文段第一个字节的序号                                             |
| 数据偏移     | 表示TCP头部长度                                                        |
| 控制位      | 包括 `SYN`（建立连接）、`ACK`（确认）、`FIN`（终止）、`RST`（复位）、`URG`（紧急）、`PSH`（推送） |
| 窗口大小     | 用于流量控制，告诉对方还能接收多少字节                                              |
| 校验和      | 检测头部和数据在传输中是否出错                                                  |
| 紧急指针     | 与 `URG` 配合，表示紧急数据的结束位置                                           |
| 选项       | 如最大报文段长度（MSS）、窗口缩放因子等                                            |

## 三. 端口

端口号是计算机网络中的一种标识，用于区分不同的网络服务和应用程序。每个端口号都是一个16位的数字，范围从0到65535。端口号分为三类：

- **知名端口号（Well-Known Ports）：** 范围从0到1023，通常分配给常见的网络服务和应用程序。
- **注册端口号（Registered Ports）：** 范围从1024到49151，通常分配给用户注册的服务和应用程序。
- **动态或私有端口号（Dynamic or Private Ports）：** 范围从49152到65535，通常用于临时或私有的连接。

### 常见的默认端口号

以下是一些常见的默认端口号及其对应的网络服务：

### 安全考虑

默认端口号的使用虽然方便，但也带来了一定的安全隐患。了解这些端口号的安全风险并采取适当的措施进行保护是非常重要的。

### HTTP 和 HTTPS

- **HTTP（超文本传输协议）：** 使用端口号80。HTTP 是用于万维网上数据传输的主要协议。
- **HTTPS（超文本传输协议安全）：** 使用端口号443。HTTPS 是 HTTP 的安全版本，通过 SSL/TLS 加密来保护数据传输的安全性。

### FTP

- **FTP（文件传输协议）：** 使用端口号20（数据传输）和21（控制）。FTP 用于在网络上传输文件，主要用于上传和下载文件。

### SMTP 和 POP3

- **SMTP（简单邮件传输协议）：** 使用端口号25。SMTP 用于发送电子邮件。
- **POP3（邮局协议第3版）：** 使用端口号110。POP3 用于从邮件服务器下载电子邮件。

### IMAP

- **IMAP（互联网邮件访问协议）：** 使用端口号143。IMAP 是另一种用于从邮件服务器读取邮件的协议，支持更高级的邮件管理功能。

### DNS

- **DNS（域名系统）：** 使用端口号53。DNS 将域名解析为IP地址，使用户能够使用人类可读的域名访问网站。

### SSH

- **SSH（安全外壳协议）：** 使用端口号22。SSH 用于加密的远程登录和其他安全网络服务。

### Telnet

- **Telnet：** 使用端口号23。Telnet 是一种早期的远程登录协议，虽然现在已被 SSH 取代，但在某些情况下仍然使用。

### RDP

- **RDP（远程桌面协议）：** 使用端口号3389。RDP 允许用户远程访问和控制另一台计算机的桌面界面。

### SNMP

- **SNMP（简单网络管理协议）：** 使用端口号161 和162。SNMP 用于网络设备的管理和监控。

### LDAP

- **LDAP（轻量目录访问协议）：** 使用端口号389。LDAP 用于访问和管理分布式目录信息服务。

2. 唯一性

公网IP：全球唯一，不可重复。

内网IP：不同局域网内可重复使用，如多个家庭网络均可使用192.168.1.1。

3. 使用场景

公网IP：用于互联网通信（如网站[服务器](https://cloud.tencent.com/product/cvm?from_column=20065&from=20065)、云服务）。

内网IP：局域网内部通信（如家庭设备、公司内部服务器）。

4. 访问权限

公网IP：可从互联网直接访问，需防火墙保护。

内网IP：无法从外网直接访问，需通过NAT（网络地址转换）或端口映射实现外网通信。

5. 获取方式

公网IP：由ISP（如电信、联通）动态或静态分配，可能需额外付费。

内网IP：由本地路由器或DHCP服务器自动分配，免费使用。

6. 成本

公网IP：因IPv4资源稀缺，静态公网IP通常收费较高。

内网IP：无额外成本，可无限复用。

7. 安全性

公网IP：暴露于互联网，易受攻击，需严格安全措施。

内网IP：受局域网保护，外部无法直接访问，相对安全。

8. 典型应用

公网IP：网站（如百度服务器）、远程访问公司系统。

内网IP：家庭NAS、打印机共享、内部文件传输。

技术原理补充

NAT（网络地址转换）：内网设备通过路由器公网IP访问外网，实现“多设备共享一个公网IP”。

IPv6差异：IPv6地址充足，理论上无需NAT，但实际中仍可能使用唯一本地地址（ULA，如fd00::/8）作为内网IP。

示例场景

访问网站：你的手机（内网IP：192.168.1.5）通过路由器（公网IP：120.230.150.10）访问百度（公网IP：39.156.66.10）。

局域网传输：办公室电脑（10.0.0.2）直接向同事电脑（10.0.0.3）发送文件，无需经过外网。

总结

公网IP是互联网的“门牌号”，内网IP是局域网的“房间号”。前者用于全球通信，后者用于内部连接，通过NAT技术实现内外网互通。理解两者的区别有助于网络配置、安全防护及故障排查。

## 物理地址 — MAC地址
由IEEE统一分配厂商代码，厂商固化在网卡中

理论上固化，但可通过软件“伪造/克隆”（修改驱动或注册表）

作用范围：
只在**同一个局域网（广播域）** 内有效，路由器不转发MAC地址跨网络

### ARP协议

- **作用**：将IP地址解析为同一局域网内的MAC地址。
    
- **为什么需要ARP**：因为网络层使用IP寻址，但最终在物理链路上传输时，网卡/交换机必须知道目标的MAC地址才能发送帧。
